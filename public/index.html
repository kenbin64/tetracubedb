◈ Data Shapes
WHAT YOU CAN STORE

The API speaks in familiar shapes — scalars, columns, rows, tables, time-series, and whole datasets. Each shape is addressed by a coordinate on the surface; the rest is up to you.

1
SCALAR
number · string · bool
A single value at one address. The atom of all data.
score = 9999
2
COLUMN
array · list
An ordered sequence of scalars sharing one logical key.
[1, 1, 2, 3, 5, 8]
3
ROW
object · record
A row of named fields keyed by a row id.
{ username, score, avatar }
4
TABLE
rows × columns
A two-dimensional set of rows under one table name within a namespace.
players, items, sessions, scores
5
TIME-SERIES
append-only event log
Per-cell history. Push events; the latest is the current state, the rest is replayable history.
score deltas · chat events · game frames
6
DATASET
whole namespace
All tables and time-series under one namespace, treated as a single addressable unit.
an entire app or game's data
7
INSTANCE
a tenant · atomic from outside
A complete tenant on the server. Other tenants see it only as an identity, never its contents.
kensgames, butterflyfx, your-app

BACKGROUND

TetracubeDB is inspired by classical work on triply periodic minimal surfaces — in particular the gyroid, discovered by Alan Schoen at NASA in 1970 (NASA TN D-5541). The gyroid is well-known and in the public domain; it shows up in butterfly wings, soap films, block-copolymer self-assembly, and countless graphics shaders.

sin(x)cos(y) + sin(y)cos(z) + sin(z)cos(x) = 0

That equation is just the gyroid's level set — textbook math, free for anyone to use. What TetracubeDB does with it is treat the surface as an addressing scheme: related cells end up close to each other in space, so spatial queries become relational queries.

On top of that the system exposes a small, conventional API surface: REST for reads and writes, WebSockets for live subscriptions, SQLite for persistence, and Bearer tokens for auth (industry standards, all of them).

The data model is also deliberately familiar: scalars, columns, rows, tables, time-series, and whole datasets. If you've used a key-value store, a document database, or an event log, you already know the shapes.

Note: this page is intentionally light on internals. The novel parts of the system — how exactly addresses are derived, how tenants are isolated cryptographically, and how the runtime ties it all together — are not documented publicly.

◈ Architecture
HOW IT WORKS
01
📐
SURFACE ADDRESSING
Each cell is assigned a coordinate on the gyroid surface so that related cells end up close together. Spatial proximity queries naturally double as relational queries.
02
🗄️
PERSISTENT CELLS
Values are persisted in SQLite outside the application directory, so a redeploy never loses data. Every cell carries its address with it.
03
⚡
SERVER-SIDE HOOKS
Register hooks against table or column patterns. Every matching write fires the hook — useful for validation, derived fields, and external side effects.
04
📡
LIVE SUBSCRIPTIONS
Clients open a WebSocket and subscribe to a table or namespace. Updates are pushed as they happen — no polling.
05
🔐
BEARER AUTH
REST and WebSocket clients authenticate with standard Bearer tokens (RFC 6750). Stored hashes use bcrypt; access is scoped per namespace.
06
🌐
MULTI-TENANT
Multiple applications share a single TetracubeDB instance via namespace isolation. Each tenant sees only its own data.
◈ Developer Reference
REST + WebSocket API
# Store a value
POST /v1/cell/:namespace/:table/:row/:col
Authorization: Bearer client_id:api_key
Content-Type: application/json

{ "value": 9999 }

# Response
{
  "ok": true,
  "addr": "…",           // opaque cell address
  "hooks": [{ "pattern": "auto-history", "ok": true }]
}
# Get a single cell
GET /v1/cell/kensgames/players/user-ken/score
Authorization: Bearer client_id:api_key

→ { "value": 9999 }

# Get an entire row
GET /v1/row/kensgames/players/user-ken

→ { "_row": "user-ken", "score": 9999, "username": "KenMaster" }

# Scan a full table
GET /v1/table/kensgames/players

# Proximity query — find related cells nearby on the surface
POST /v1/query/near
{ "namespace": "kensgames", "of": "players/user-ken/score", "radius": 2.0 }
# Append an event to a cell's history (time-series)
POST /v1/stack/kensgames/score_history/user-ken/score
{ "delta": { "score": 12500, "ts": 1745001000 } }

→ { "ok": true, "seq": 2 }

# Read the full event series
GET /v1/stack/kensgames/score_history/user-ken/score

→ {
    "frames": [
      { "seq": 1, "delta": { "score": 9999 },  "at": 1776787769 },
      { "seq": 2, "delta": { "score": 12500 }, "at": 1776787770 }
    ]
  }
// Connect WebSocket
const ws = new WebSocket('wss://tetracubedb.com/ws');

ws.onopen = () => {
  // Authenticate
  ws.send(JSON.stringify({
    type: 'auth',
    client_id: 'your-client-id',
    api_key:   'your-api-key'
  }));
};

ws.onmessage = e => {
  const msg = JSON.parse(e.data);

  if (msg.type === 'auth_ok') {
    // Subscribe to a table
    ws.send(JSON.stringify({ type: 'subscribe', table: 'kensgames:players' }));
  }

  if (msg.type === 'set') {
    console.log('live update', msg.row, msg.col, msg.value);
  }
};
// Node.js or browser UMD client
const { TetracubeClient } = require('./tetracube_client');

const db = new TetracubeClient({
  url:      'https://tetracubedb.com',
  clientId: 'your-client-id',
  apiKey:   'your-api-key',
  namespace:'myapp'
});

// Write a value
await db.set('users', 'alice', 'score', 4200);

// Read a row
const player = await db.getRow('users', 'alice');

// Append to a cell's event series
await db.pushDelta('events', 'alice', 'action', { move: 'jump', t: Date.now() });

// Real-time subscription
db.subscribe('users', ({ row, col, value }) => console.log(row, col, value));
await db.connectWS();
